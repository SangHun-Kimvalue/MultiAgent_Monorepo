# 로드맵 문서 템플릿

페이즈가 2개 이상이거나 다중 세션/다중 도구 협업이면 리포에 둔다. 다음 세션의 cold-start 비용을 줄이는 핵심 핸드오프
(채팅 맥락은 세션을 못 넘지만 파일은 넘는다). 아래 구조를 채운다.

```markdown
# <기능> — 전체 페이즈 로드맵

작성일: <date>
문서 성격: 다중 세션 협업용 진행 기준(SoT 보조).
대상/전제: <적용 대상 + 핵심 전제>
상위 SoT: <원본 설계 문서들 + §절>

## 0. 확정 결정 로그 (변경 시 이 표를 갱신)
| 항목 | 확정값 | 근거 |
|---|---|---|
| <보류였던 결정들: 저장 위치 / 스레드 모델 / id 포맷 / 전략 …> | | |

## 1. 페이즈 개요
| Phase | 내용 | 상태 |
|---|---|---|
| A | … | ✅ 완료 / 🔄 진행 / ⬜ 예정 |

## 2. 완료 산출물 (기준 코드) — 공개 API(변경 금지) 명시

## 3. 다음 Phase 상세 — 부착점(file:line 실측) 표 + 소유/수명주기 + 스레드 안전 + DoD

## 4~ . 후속 Phase 예고 — 증분(C1→C2…)으로 쪼개서

## 끝. 불변 원칙 (전 페이즈 공통)
```

## 원칙
- **결정 로그가 핵심.** "나중에 결정"한 항목을 여기서 확정값으로 못박는다. 리뷰가 전략을 흔들면 여기를 근거로 거부/반영.
- **부착점은 file:line + 시그니처.** "라인 이동 가능 → 재확인" 명시.
- **상태 컬럼을 갱신.** 완료/진행/예정을 최신으로.
- **큰 페이즈는 증분으로.** "evidence 보강"을 한 줄로 두지 말고 C1→C2→C3→C4 순서로.
- **수집형 작업이면 "수집 능력표"와 "구현 우선순위"를 분리.** 능력표(무엇을 모을 수 있나) ≠ 우선순위(언제 구현/무엇을
  끝까지 지킬지). 프로젝트 전략(collect-max prune vs minimal-first)을 명시.
