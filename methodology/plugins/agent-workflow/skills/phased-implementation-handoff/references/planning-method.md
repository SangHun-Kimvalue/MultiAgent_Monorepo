# 계획 수립 방법 (해석·검증 상세)

Planner 세션이 "정확한 핸드오프"를 만드는 절차. 핵심은 **추정 금지 + 실측 + 명시적 결정**.

## 1. SoT(Source of Truth) 독해
- 설계 문서가 여러 개면, **권위 스펙 문서**와 **alias/역할명**을 구분한다. 같은 책임이 문서마다 다른 클래스명으로
  등장하는 함정이 흔하다(예: 역할명 vs 실제 1차 구현 클래스명). 권위표/정합성 리뷰 섹션을 우선한다.
- "이번 범위"와 "보류/후속"을 문서에서 먼저 분리한다.

## 2. 코드 실측 (가장 중요 — 환각 방지)
- 부착점·시그니처·진입 경로를 `Grep`/`Read`로 **file:line** 확인한다. 절대 기억/추정으로 적지 않는다.
- 오버로드/중복 진입점에 주의(예: 같은 이름 함수 2개 → 실제 진입 경로 하나에만 부착, 중복 호출 방지).
- 호출 스레드를 확인한다(콜백/WebSocket 백그라운드 스레드면 비차단·UI 금지가 불변식이 된다).
- 라인은 이동하므로 프롬프트엔 "시그니처로 재확인 후 부착"을 함께 적는다.

## 3. 컨벤션 체크리스트
빌드가 깨지거나 리뷰에서 막히는 단골 지점:
- **싱글톤/수명주기 패턴**(GetInstance/Init/UnInit/ReleaseInstance 등) 일치.
- **JSON/직렬화 라이브러리** 기존 것 재사용.
- **경로**는 PathManager류 경유(하드코딩 금지), 저장 위치는 application-owned 영역.
- **빌드 특이점**: 강제 선행 include(pch), 플랫폼 전용 헤더 → 신규 파일은 자기완결(독립 컴파일 가능)하게.
- **스레드 안전**: 콜백 스레드에서 파일 I/O·UI·블로킹 금지. bounded queue/async writer로 분리.

## 4. 보류 항목 명시적 확정
"착수 시 결정"으로 남은 항목(저장 루트, 스레드 모델, id 포맷, 수집 전략 등)을 **이 계획 단계에서 값으로 확정**하고
로드맵 결정 로그에 박는다. 흐릿하게 두면 구현자가 제각각 해석한다.

## 5. 토큰 쓰기 전 선행 점검 (스킬 §2와 동일, 반드시)
- 다음 페이즈의 **전제 충족 여부**를 확인. 모르면 **생성 전에 질문**한다.
- scope가 갈리면(이 페이즈만/묶기, 범용/특정) 생성 전에 질문한다.
- 이미 진행 중 세션의 산출물이 있으면 "있으면 재생성 말고 정렬/완성" 지시.

## 6. 외부 리뷰 비판적 필터
Reviewer 세션/Nitpicker/사람 리뷰를 받을 때:
- 좋은 지적도 **프로젝트 전략과 충돌하면 거부 또는 재해석**한다(예: "수집 줄여라" 리뷰가 collect-max 전략과 반대면 반영 금지).
- 진행 중 구현과 충돌하는 walk-back은 churn이므로, "제거" 대신 "기본값/옵션으로 완화" 같은 절충을 우선.
- 무엇을 반영/거부했는지 사용자에게 명확히 보고.
