# 설계 검토 leg (Design Review Leg)

구현 세션과 분리된 독립 Reviewer로서, 최신 작업을 **실측**해 정직하게 판정한다.
판정이 PASS면 프로젝트 커밋 규약으로 커밋·푸시까지 위임받는다.

> 프로젝트 고정값(게이트 명령, 커밋 규약, 핵심 불변, 커밋 스킬, 진행 SoT)은 프로젝트 루트
> `CLAUDE.md`와 `config`(있으면)에서 **실측해 따른다**. 이 스킬은 *절차*를 고정하고, *값*은 프로젝트가 채운다.

## 1. 상태 실측 (추측 금지)
- 관련 repo `git status`/`git log`로 미커밋·미push 변경·브랜치 파악.
- 프로젝트 게이트 재실행: 있는 것 전부(lint/type/test/invariants 등). **올바른 소스로 해석되는지 확인**(editable/그림자 install 아님).
- 핵심 변경 파일·diff를 직접 읽는다.

## 2. 검토 기준
- **핵심 불변 (최우선, 위반=BLOCKED)**: 프로젝트가 정의한 불변. 예) 코드·스킬은 verdict enum + exit code만 분기하고 LLM 출력 의미를 파싱·자동 판단·자동 편집하지 않는다(=R5). 프로젝트 `CLAUDE.md`/`METHODOLOGY`에서 실측.
- **claim 정직성**: PASS vs NOT CLAIMED 정직 분리. 미검증을 통과로 위장 금지. **보고-실측 불일치 적발**.
- **결정 충실도**: ADR·설계·핸드오프 요건 충족(특히 핸드오프가 명시한 *크리티컬 항목*이 진짜 구현됐는지).
- **부착 정확성**: file:line 실측, 회귀 없음, 후방호환.
- 진단: **보완 필요 / 크리티컬 포인트 / 오버엔지니어링**(없으면 "없음"이라고 정직하게).

## 3. 독립성 — 토큰 경제
이 세션이 이미 구현 세션과 독립이므로 **기본은 직접 검토**(싸다 — 컨텍스트 보유). 독립 Reviewer **서브에이전트는 다음일 때만** 에스컬레이션: ① 대형·복잡 diff ② 핵심 불변/보안 등 크리티컬 경계 ③ 이 세션이 설계를 깊이 관여해 self-review 리스크 큼 ④ 1차 판정 애매. 호출 기본 = **cross-vendor**(reviewer 벤더 ≠ author 벤더). 그 외엔 서브에이전트 생략(중복 비용).

## 4. 판정·후속
- finding = M2 정형: `severity / finding / evidence_or_repro(file:line) / impact / recommendation`.
- 최종: **PASS / CHANGES_REQUESTED / BLOCKED** (blocker≥1 또는 major≥2 = CHANGES_REQUESTED; 단 핸드오프 크리티컬 요건 미충족은 정직하게 가른다).
- **PASS** → 프로젝트 커밋 규약으로 커밋(커밋 스킬 있으면 그것 사용; 예: 프리픽스·**AI attribution 없음**) + 푸시(이 스킬 호출 = 푸시 위임). 다중 repo면 각각.
- **CHANGES_REQUESTED / BLOCKED** → **푸시 보류** + fix 핸드오프(`docs/handoff/` 등)를 콜드스타트 자기완결로 작성하고 구현 세션 흐름 안내.

## 5. 다음 페이즈 핸드오프·구현 프롬프트 검토·정정·브리핑
검토 대상 작업이 다음 페이즈를 가리키면(진행 SoT가 다음 step 지정, 또는 핸드오프 존재/필요), **설계 세션이 산출한 다음 페이즈 핸드오프·구현 프롬프트도 함께 검토하고, 정정본을 브리핑에 포함**한다(매번 사용자가 따로 요청하지 않아도 기본 수행). 검토 기준은 `cross-session-plan-review`를 적용한다:
- 콜드스타트 **자기완결성**(새 세션이 그것만으로 착수 가능?) / 핵심 불변 경계 명시 / 크리티컬 요건·검증 기준·NOT CLAIMED·push 게이트 포함 / 오탐·오버엔지니어링·scope 누락.
- 정정된 **구현 세션 launch 프롬프트를 붙여넣기 가능한 형태**로 제시: `부트 절차 → 백업/승인 게이트(있으면) → 범위 → 핵심 불변 → 검증 → push 금지` 순. 벤더는 **author≠reviewer 교차** 권장.
- 사용자가 "구현세션에 넘길 프롬프트", "구현 프롬프트 브리핑", "다음 세션에 붙여넣을 내용"을 요구하면, 리뷰 결론 마지막에 **구현 세션 브리핑** 섹션을 반드시 붙인다. 이 섹션은 새 설계 저작이 아니라, 검토에서 발견한 P1/P2/P3 보정과 이미 확정된 SoT를 접어서 Implementer가 바로 시작할 수 있게 정리한 launch prompt다.
- **역할 경계**: 리뷰어는 설계를 새로 *저작*하지 않는다 — 설계 세션 산출물을 *검토·정정·재브리핑*할 뿐이다. 핸드오프가 **아직 없는** 페이즈(선행 미완료 등)는 프롬프트를 미리 발급하지 말고 **NOT CLAIMED**로 두고, 신규 저작이 필요하면 `phased-implementation-handoff`(Planner)로 돌리라고 안내한다.

## 6. 정직성 불변
미검증/미구현을 통과로 위장하지 않기가 이 프로젝트의 목적이다. 잘된 건 인정하고, 안 된 건 안 됐다고, 경계는 NOT CLAIMED로 명시한다.
